# How It Works

A step-by-step walkthrough of AI Footprint's detection, attribution, and provenance pipeline.

---

## Step 1: Register known AI snippets

```bash
ai-footprint init
ai-footprint add-snippet --file copilot-output.ts --source "copilot" --model "gpt-4.1"
```

When you register a snippet:

1. The code is **normalized** — trailing whitespace stripped, line endings unified, content trimmed.
2. A **SHA-256 hash** is computed from the normalized content.
3. The snippet record (hash, source, model, tool, timestamp, original content) is appended to `~/.ai-footprint/snippets.json`.

```json
{
  "id": "a1b2c3d4-...",
  "hash": "e3b0c44298fc1c149afbf4c8996fb924...",
  "source": "copilot",
  "model": "gpt-4.1",
  "tool": "copilot",
  "addedAt": "2026-03-31T10:00:00.000Z",
  "content": "function fibonacci(n) { ... }"
}
```

## Step 2: Scan your codebase

```bash
ai-footprint scan ./src
```

The scanner walks every code file (22+ supported extensions), skipping `node_modules`, `.git`, `dist`, etc.

For each file, three detection methods run in order:

### 2a. Exact matching

A sliding window (sized to each registered snippet) moves line-by-line through the file. Each window is normalized and hashed. If the hash matches a known snippet, it's an **exact match** (confidence: `high`, similarity: `100%`).

```
File:       line 1  line 2  line 3  line 4  line 5  line 6
                    ├────── window (3 lines) ──────┤
                    hash("line 2\nline 3\nline 4") === snippet.hash?
```

### 2b. Fuzzy matching

For snippets that have been lightly edited (variable renames, reformatting, minor additions), fuzzy matching catches what exact hashing would miss.

**Token n-grams:**
```
Original:  function fetchData(url) { ... }
Tokens:    ["function", "fetchData", "(", "url", ")", "{", "...", "}"]
3-grams:   {"function fetchData (", "fetchData ( url", "( url )", ...}
```

**Structural n-grams** (variable-agnostic):
```
Original:  function fetchData(url) { ... }
Structural: ["function", "ID", "(", "ID", ")", "{", "...", "}"]
3-grams:    {"function ID (", "ID ( ID", "( ID )", ...}
```

The scanner computes **Jaccard similarity** (intersection ÷ union) for both token and structural shingle sets, then combines:

```
combined = 0.4 × token_similarity + 0.6 × structural_similarity
```

If `combined ≥ threshold` (default 0.6), it's a **fuzzy match**. Score ≥ 0.85 → `high` confidence, otherwise `medium`.

### 2c. Pattern matching

Regex heuristics scan each line for known AI markers:

| Pattern | Detects |
|---------|---------|
| `// generated by gpt\|copilot\|claude\|...` | Inline comments |
| `/* @ai-generated */` | JSDoc markers |
| `# generated by ...` | Python/Ruby comments |
| `AI-generated`, `ai_generated` | Generic markers |
| `copilot`, `GitHub Copilot` | Tool references |

Pattern matches are confidence `medium`, match type `pattern`.

## Step 3: Git hook attribution

When git hooks are installed:

```bash
cp git-hooks/pre-commit  .git/hooks/pre-commit
cp git-hooks/commit-msg  .git/hooks/commit-msg
chmod +x .git/hooks/pre-commit .git/hooks/commit-msg
```

### On `git commit`:

1. **pre-commit hook** scans `git diff --cached` for AI patterns and known snippets. Prints a warning if matches are found.

2. **commit-msg hook** re-scans the staged diff. If matches are found, it runs:
   ```
   git interpret-trailers --in-place --trailer "AI-Footprint: 3 match(es); model(s): gpt-4.1" .git/COMMIT_EDITMSG
   ```

3. The commit is saved with the trailer embedded in the message:
   ```
   add helper functions

   AI-Footprint: 3 match(es); model(s): gpt-4.1
   ```

### Querying provenance

The trailer lives inside Git — no external system required:

```bash
# Find all commits with AI attribution
git log --grep="AI-Footprint:"

# Count AI-attributed commits
git log --oneline --grep="AI-Footprint:" | wc -l

# Show full provenance for a range
git log --format="%H %s %b" --grep="AI-Footprint:" v0.1..HEAD
```

## Step 4: CI enforcement (GitHub Action)

```yaml
- uses: ./action
  with:
    fail-on-unattributed: "true"
    fail-threshold: "5"
```

The action runs the full scan pipeline in CI:

1. **Scan** — analyzes all code in the PR diff
2. **Annotate** — posts `::warning` / `::notice` annotations on matched lines (visible in the PR "Files changed" tab)
3. **Comment** — posts a summary table on the PR with file, line, type, confidence, source, and similarity
4. **Enforce** — optionally fails the check if unattributed AI code exceeds threshold

## Step 5: Visualization (Dashboard)

```bash
ai-footprint dashboard
```

Each scan is persisted to `~/.ai-footprint/history/scans.json` with git metadata (repo, branch, commit hash, timestamp). The dashboard reads this history and renders:

- **Stat cards** — files analyzed, AI-attributed, suspicious, top model
- **Trend chart** — AI code share over time (bar chart, last 30 scans)
- **Repo cards** — per-repo summaries with sparkline trends
- **Match table** — latest matches with type/confidence/similarity badges

## Detection accuracy

| Method | Catches | Misses |
|--------|---------|--------|
| Exact hash | Byte-identical AI code | Any modification |
| Fuzzy (≥60%) | Renamed variables, reformatted, minor edits | Major rewrites |
| Fuzzy (≥85%) | Minor whitespace/naming changes | Structural changes |
| Pattern regex | Explicit AI markers in comments | Removed/absent markers |

For maximum coverage, use all three methods together (enabled by default).
