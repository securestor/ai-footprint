# AI Footprint

**Git-native provenance tracking for AI-generated code.**

Track, attribute, and audit AI-generated code directly in your Git workflow.

---

## The problem

AI-generated code is everywhere — but invisible.

Copilot, ChatGPT, Claude, and dozens of other tools produce code that lands in your repos with zero attribution. You can't tell what was written by a human, what was generated, or which model produced it. There's no audit trail, no provenance, no accountability.

Your Git history says *who committed it*. It says nothing about *what wrote it*.

## The solution

AI Footprint tracks AI-generated code directly in Git.

It hashes known AI snippets, scans your code for matches and heuristic patterns, and attaches provenance metadata to commits automatically — using Git's own trailers as the source of truth. No external database, no SaaS, no lock-in.

## Demo

```
$ git add utils.ts
$ git commit -m "add helper functions"

[ai-footprint] Detected AI code — 2 known snippet(s); 1 pattern match(es); model(s): gpt-4.1
[ai-footprint] Added trailer → AI-Footprint: 3 match(es); model(s): gpt-4.1
```

Your commit message now includes:

```
add helper functions

AI-Footprint: 3 match(es); model(s): gpt-4.1
```

Every commit carries its own provenance. Queryable with `git log --grep`, no extra tooling needed.

## Quickstart

```bash
npm install -g ai-footprint

# Initialise registry + auto-install git hooks
ai-footprint init
```

That's it — `init` creates the snippet registry at `~/.ai-footprint/snippets.json` and installs `pre-commit` / `commit-msg` hooks into `.git/hooks/` automatically.

Register a known AI snippet:

```bash
ai-footprint add-snippet --file snippet.ts --source "chatgpt" --model "gpt-4.1"
```

Bulk-register an entire directory:

```bash
ai-footprint track ./src --source "copilot" --model "gpt-4.1"
```

Scan your repo:

```bash
ai-footprint scan
```


## How it works

### 1. Snippet hashing

When you register a snippet, AI Footprint normalizes whitespace and computes a SHA-256 hash. This fingerprint is stored in a local registry at `~/.ai-footprint/snippets.json`.

### 2. Matching

On scan (or commit), AI Footprint checks your code five ways:

- **Exact match** — sliding-window hash comparison against all registered snippets
- **Fuzzy match** — n-gram shingling with Jaccard + structural similarity catches code that's been renamed, reformatted, or lightly edited (configurable threshold, default 60%). Uses pre-tokenised line caching and early-exit heuristics for fast scanning.
- **AST-level match** — extracts structural features (function signatures, control flow, class hierarchies, imports) using language-aware heuristic parsing, then compares abstract syntax shapes via AST-fingerprint shingling + LCS sequence similarity. Catches deep refactors where variable names, formatting, and comments have all changed. Supports TypeScript/JavaScript, Python, Go, and shell scripts.
- **Tree-sitter native match** *(optional)* — uses the `tree-sitter` native binary to build real parse trees, then compares subtree structures for even deeper refactor detection. Combines shingle Jaccard (35%) + LCS sequence (35%) + subtree similarity (30%). Falls back automatically to the regex-based AST matcher when tree-sitter is not installed.
- **Heuristic patterns** — regex detection of AI markers (`// generated by`, `@ai-generated`, `copilot`, etc.)

Matches are tagged with a confidence level: `high` (exact hash or ≥85% similarity), `medium` (pattern or ≥60% similarity).

### 3. Git binding

Git hooks intercept each commit and scan the staged diff. When AI code is detected, a structured `AI-Footprint:` trailer is appended to the commit message via `git interpret-trailers`. The provenance lives *inside* Git — portable, grep-able, and tamper-evident.

## CLI commands

| Command | Description |
|---|---|
| `ai-footprint init` | Create the snippet registry + install git hooks |
| `ai-footprint add-snippet` | Register a known AI-generated snippet |
| `ai-footprint track [dir]` | Bulk-register all code files in a directory |
| `ai-footprint scan [dir]` | Scan a directory for AI-generated code |
| `ai-footprint report [dir]` | Alias for `scan` with detailed output |
| `ai-footprint sbom [dir]` | Export an SBOM (CycloneDX 1.5 or SPDX 2.3) |
| `ai-footprint team <cmd>` | Manage the shared team registry |
| `ai-footprint intercept` | Start the LLM API interception proxy |
| `ai-footprint intercept-status` | Check proxy status |
| `ai-footprint treesitter` | Show tree-sitter native support status |
| `ai-footprint audit` | Verify audit log integrity (hash chain) |
| `ai-footprint security` | Harden permissions + verify registry signature |
| `ai-footprint dashboard` | Launch the web dashboard (default port 3120) |
| `ai-footprint hook` | Run as a git hook (used internally by hooks) |

### Options for `add-snippet` / `track`

| Flag | Description |
|---|---|
| `--file <path>` | Single file to register |
| `--dir <path>` | Directory to register (all code files) |
| `--source <src>` | Origin of the code (e.g. `ai`, `chatgpt`, `human`) — **required** |
| `--model <name>` | AI model name (e.g. `gpt-4.1`, `claude-opus-4.6`) |
| `--tool <name>` | Tool used (e.g. `copilot`, `cursor`) |

## GitHub Action

Add to your workflow to get CI-level provenance checks on every PR:

```yaml
- name: AI Footprint Scan
  uses: ./action
  with:
    scan-path: "."
    fuzzy-threshold: "0.6"
    fail-on-unattributed: "false"
    post-comment: "true"
```

The action will:
- Scan the PR diff for AI-generated code
- Post inline annotations on matched lines
- Comment a summary table on the PR
- Optionally fail the check if unattributed AI code exceeds a threshold

## Team Registry

Share snippet databases across your team — two modes:

### Git-backed (recommended)

Store the shared registry in a Git repo. Every team member pulls/pushes snippets like code.

```bash
# Configure (once per machine)
ai-footprint team config --git-url git@github.com:myorg/ai-registry.git --team backend

# Pull shared snippets into your local registry
ai-footprint team pull

# Push your local snippets to the team
ai-footprint team push

# Check status
ai-footprint team status
```

### API-backed

Connect to a centralised registry server.

```bash
ai-footprint team config --api-url https://registry.example.com --api-key $TOKEN --team backend
ai-footprint team pull
ai-footprint team push
```

The `pull` command merges remote snippets into your local `~/.ai-footprint/snippets.json` (deduplicated by hash). The `push` command uploads your local snippets to the shared store.

## SBOM Integration

Export AI provenance data to industry-standard Software Bill of Materials formats:

```bash
# CycloneDX 1.5 (default)
ai-footprint sbom --format cyclonedx --output bom.json

# SPDX 2.3
ai-footprint sbom --format spdx --output spdx.json

# Scan a specific directory
ai-footprint sbom ./src --format cyclonedx --output src-bom.json
```

Each AI-attributed file becomes a component/package with provenance metadata:

- **CycloneDX** — components with `ai-footprint:*` properties (model, source, confidence, similarity, matchType)
- **SPDX** — packages with REVIEW annotations containing provenance details

Integrate with your existing SBOM toolchain, compliance checks, or supply chain security pipeline.

## Tree-sitter Native

For even deeper AST analysis, install the optional tree-sitter native binary:

```bash
npm install tree-sitter tree-sitter-typescript tree-sitter-python tree-sitter-go tree-sitter-bash
```

When tree-sitter is available, the matching pipeline upgrades from regex-based heuristic parsing to real parse trees, enabling:

- **Subtree comparison** — compares actual function/class subtree structures, not just surface patterns
- **Deeper refactor detection** — catches changes that alter AST structure (e.g. extracting a function, inlining a class)
- **Multi-signal scoring** — 35% shingle Jaccard + 35% LCS sequence + 30% subtree similarity

Check your installation status:

```bash
ai-footprint treesitter
```

Tree-sitter is **fully optional** — if not installed, the scanner falls back to the regex-based AST matcher automatically.

## LLM API Auto-Detect

Intercept LLM API calls to auto-register generated code as snippets:

```bash
# Start the interception proxy
ai-footprint intercept --port 8990 --verbose
```

Configure your LLM client to route through the proxy:

```bash
# OpenAI
OPENAI_BASE_URL=http://localhost:8990

# Anthropic
ANTHROPIC_BASE_URL=http://localhost:8990

# Any HTTP client
HTTP_PROXY=http://localhost:8990
HTTPS_PROXY=http://localhost:8990
```

The proxy transparently forwards all requests to the real API, reads responses, extracts code blocks from markdown-formatted output, and auto-registers them as snippets in your local registry.

Supported providers:
- **OpenAI** — Chat Completions, Completions (legacy)
- **Anthropic** — Messages API
- **Google AI** — Gemini generateContent
- **Azure OpenAI** — Chat Completions
- **Ollama** — generate, chat
- **Cohere** — Chat
- **Mistral** — Chat Completions

Check proxy status while running:

```bash
ai-footprint intercept-status
# Or visit http://localhost:8991/status
```

The proxy applies a **default host allowlist** (OpenAI, Anthropic, Google AI, Azure OpenAI, Ollama, Cohere, Mistral, localhost) to prevent SSRF. Override with `--allowed-hosts api.custom.com,other.host`.

## Security & Threat Model

AI Footprint includes enterprise-grade security controls aligned with the **EU Cyber Resilience Act (CRA)**:

### Registry integrity

Every snippet registry can be **signed** (HMAC-SHA256) and **verified** against a locally generated 256-bit signing key (`~/.ai-footprint/signing-key`):

```bash
# Full security status — permissions, audit log, registry signature
ai-footprint security
```

### Tamper-evident audit log

A **hash-chained, append-only** audit log records all security-relevant operations — snippet additions, scans, team syncs, SBOM exports, proxy interceptions, permission fixes, and blocked requests:

```bash
# Verify the hash chain is intact
ai-footprint audit
```

Each log entry contains a SHA-256 hash linking to the previous entry. Any tampering breaks the chain and is immediately detectable.

### Input validation

| Input | Validation |
|---|---|
| Git URLs | Scheme allowlist (https, git, ssh, SCP) + control char rejection |
| API URLs | HTTPS enforced (localhost exempt) |
| Team names | Alphanumeric + hyphens/underscores, max 128 chars |
| Port numbers | Integer 1–65535 |
| Output paths | Confined within CWD (prevents path traversal) |
| Network data | Snippet schema validation, prototype pollution guard |

### Command injection prevention

All shell commands use `execFileSync` with **argument arrays** — no string interpolation into shells. Model names, team names, file paths, and git URLs are never template-interpolated into `execSync` calls.

### File permission hardening

Sensitive files (`signing-key`, `team.json`, `snippets.json`, `audit.log`) are automatically set to `0600` (owner-only read/write). The config directory uses `0700`.

### Proxy security

- **Default host allowlist** — only known LLM API endpoints reachable (prevents open relay / SSRF)
- **Hop-by-hop header stripping** — prevents request smuggling
- **Body size limits** — 50 MB max to prevent OOM
- **Bind to localhost only** — not exposed to network

### Symlink protection

Directory walkers (scanner, registry) skip symbolic links to prevent traversal into unexpected directories.

### CRA Compliance Matrix

| CRA Requirement | Implementation |
|---|---|
| Art. 10(1) — Security by design | Signed registry, hash-chained audit, input validation at all boundaries |
| Art. 11 — Vulnerability handling | Tamper-evident audit trail, `ai-footprint audit` command |
| Annex I §2.1 — Data protection | Control char rejection, path confinement, HTTPS enforcement |
| Annex I §2.2 — Availability | File/body size limits, graceful degradation (tree-sitter optional) |
| Annex I §2.5 — Integrity | SHA-256 fingerprints, HMAC-SHA256 signatures, hash-chained logs |
| Annex I §2.6 — Secure defaults | Host allowlist on, strict permissions, no open proxy |
| Art. 10(7) — SBOM provision | CycloneDX 1.5 + SPDX 2.3 with AI provenance |

Full threat model: [docs/THREAT-MODEL.md](docs/THREAT-MODEL.md)
